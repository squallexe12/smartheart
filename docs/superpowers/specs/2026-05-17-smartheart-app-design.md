## 1. Purpose

Ship a heart disease risk calculator app: a premium, modern web frontend backed by a from-scratch NumPy logistic regression model trained on the UCI Heart Disease (Cleveland) dataset.

The app must look and feel like a polished product ‚Äî not a notebook export, not a default-styled framework demo ‚Äî while preserving the project's core constraint that the learning algorithm is implemented by hand in NumPy.

## 2. Scope

### In scope (v1)
- NumPy-only `LogisticRegressionScratch` model
- Manual preprocessing utilities (standardization, train/test split, one-hot)
- Manual metrics (accuracy, precision, recall, F1, confusion matrix, ROC, AUC)
- Pytest suite covering the core library
- Training script that produces:
  - A pickled Python model artifact
  - A JSON weights artifact for browser-side inference
- A static, single-page web frontend that loads the JSON artifact and runs inference in the browser
- README documenting motivation, usage, and results
- `requirements.txt`

### Out of scope (v1, deferred)
- Jupyter notebook series (EDA, preprocessing, comparison, multi-model benchmark)
- Streamlit demo
- LaTeX gradient derivation doc
- CI workflows
- Regularization, momentum, mini-batch SGD
- SHAP-based explainability beyond the linear contribution breakdown
- Multi-class extension

These can be layered on later without rework to the v1 surface.

## 3. Architecture

Two cleanly separated layers:


## 5. ML Core Details

### `LogisticRegressionScratch`
- Vanilla batch gradient descent
- Sigmoid clipped to `[-500, 500]` before `exp`
- BCE loss with `epsilon = 1e-15` inside `log`
- Vectorized ‚Äî no sample-wise loops
- Tracks `cost_history` per iteration
- Public surface matches the API in CLAUDE.md ¬ß7

### Preprocessing
- `train_test_split_manual` ‚Äî stratified, seeded
- `standardize_fit` returns `(mean, std)`; `standardize_apply` uses those
- `one_hot_encode` ‚Äî drops first category to avoid collinearity, returns column names

### Metrics
- All implemented manually; cross-checked against sklearn in tests (sklearn allowed in test files only, as a reference oracle)

## 6. Training Script (`scripts/train.py`)

1. Load `data/raw/heart.csv`
2. Split into `X`, `y`
3. `train_test_split_manual` (80/20, stratified, seed 42)
4. Fit standardizer on train numerics; apply to both splits
5. One-hot encode train categoricals; align test columns
6. Save processed splits to `data/processed/`
7. Train `LogisticRegressionScratch(learning_rate=0.01, n_iters=1000)`
8. Evaluate on test set; collect metrics
9. Persist `models/scratch_model.pkl`
10. Write `web/public/model.json` per the contract in ¬ß4
11. Print a results table

## 7. Web Frontend Details

### Stack
- HTML5
- Tailwind via CDN (no build step ‚Äî keeps the project portable and frictionless)
- Vanilla JS (no React/Vue) ‚Äî the app is small enough that a framework adds cost without benefit
- Google Fonts: `Inter` for UI, `JetBrains Mono` for numbers

### Visual direction
- Dark premium UI: deep slate/zinc background (`#0b0f17`-ish), layered surfaces (raised cards on a subtle gradient base), soft inner highlights
- Accent palette:
  - **Risk red** ‚Äî used sparingly, only for high-risk states and the result gauge top half
  - **Calm emerald** ‚Äî low-risk states, success states
  - **Neutral slate** ‚Äî borders, dividers, body text
  - **Soft amber** ‚Äî the disclaimer band
- Typography: tight tracking on display headings, comfortable leading on body, monospaced for numbers in the gauge and contribution breakdown
- Motion: only `transform` and `opacity`; gauge fill animates; result card fades and lifts in; no `transition-all`
- Every interactive control has hover, focus-visible, and active states
- Custom controls ‚Äî sliders for numeric inputs styled with custom track/thumb; segmented buttons for binary categoricals; styled selects for multi-value categoricals
- No generic Tailwind defaults ‚Äî no default blue, no flat shadows, no untouched form widgets

### Page structure (single page)
1. **Hero** ‚Äî short, confident pitch + "Try it" CTA scrolling to the calculator
2. **Calculator** ‚Äî the main attraction
   - 13 inputs grouped into "Demographics", "Vital signs", "Cardiac signals"
   - Sensible defaults pre-filled (median values from training data)
   - Real-time prediction as the user changes inputs (debounced)
   - Result panel: animated risk gauge (0‚Äì100%), risk band label ("Low / Moderate / Elevated / High"), top 3 contributing factors with direction
3. **How it works** ‚Äî short explainer with the math (sigmoid + dot product + threshold) shown as actual rendered math (KaTeX via CDN) or a clean code block fallback
4. **About the model** ‚Äî dataset, accuracy badge, link to GitHub repo
5. **Disclaimer** ‚Äî prominent: "Academic project. Not a medical device. Not a substitute for clinical judgment."
6. **Footer** ‚Äî author, GitHub link, year

### Inference logic
```js
function predict(rawInputs, model) {
  const x = encode(rawInputs, model.preprocessing);     // one-hot + ordering
  const z = standardize(x, model.preprocessing);         // (x - mean) / std on numerics only
  const score = dot(z, model.weights) + model.bias;
  const proba = 1 / (1 + Math.exp(-clip(score, -500, 500)));
  const contributions = z.map((v, i) => v * model.weights[i]);
  return { proba, contributions };
}
```

Top contributions (by absolute value, signed) feed the "what's pushing your risk up/down" panel.

## 8. Testing Strategy

- `test_model.py`
  - `sigmoid(0) == 0.5`
  - Sigmoid handles `¬±1e6` without overflow
  - `fit` reduces cost monotonically over the first N iterations on a separable toy dataset
  - `predict_proba` outputs always in `[0, 1]`
  - `predict` returns `0` or `1` only
  - Reproducibility: same seed ‚Üí identical weights
- `test_preprocessing.py`
  - `standardize_apply(standardize_fit(X), X)` produces zero mean, unit variance per column
  - `train_test_split_manual` preserves class ratios within tolerance
  - `one_hot_encode` shape and content match expected
- `test_metrics.py`
  - Manual `accuracy`, `precision`, `recall`, `f1_score` match `sklearn.metrics` equivalents on the same inputs (sklearn allowed only here as oracle)
  - `confusion_matrix` shape `(2, 2)` and orientation correct
  - `auc` matches `sklearn.metrics.roc_auc_score`

## 9. Success Criteria

- `pytest tests/ -v` passes
- `python scripts/train.py` completes, prints metrics, writes both artifacts
- Trained from-scratch model ‚â• 75% test accuracy
- `web/index.html` opens in a browser, loads `model.json`, and produces a prediction on input change
- Visual design feels premium: no generic Tailwind default look; layered depth; intentional typography; restrained motion; all states styled
- README explains motivation, math, results, how to run, and links to the live demo placeholder

## 10. Risks & Mitigations

- **Accuracy below 75% threshold** ‚Äî vanilla GD without regularization may underperform. Mitigation: tune `learning_rate` and `n_iters` via training curve; if still short, allow a sensible default change before failing the gate.
- **Browser/Python prediction drift** ‚Äî easy to misalign feature ordering. Mitigation: artifact contract pins `encoded_feature_order`; add a sanity test that JS and Python produce identical predictions on a fixed input vector (manual cross-check during integration).
- **Design drifting generic** ‚Äî the easy failure mode. Mitigation: frontend-design skill is mandatory; explicit anti-generic guardrails from project CLAUDE.md apply.

## 12. Out-of-Scope Decisions Already Made

- **Streamlit dropped** in favor of the static web frontend ‚Äî better visual control, simpler deploy story (any static host).
- **No backend** ‚Äî inference runs client-side. Model weights are public (which is fine; this is an academic demo).
- **No build tooling for the frontend** ‚Äî Tailwind CDN + vanilla JS. Keeps the surface tiny and lets the user open `index.html` directly.
